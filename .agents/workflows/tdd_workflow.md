# 효율적인 TDD (Test-Driven Development) 워크플로우

이 워크플로우는 이커머스 대시보드 프로젝트에서 기능 개발 및 코드 수정 시 반드시 따라야 하는 TDD 기반 개발 절차를 정의합니다. 견고하고 안정적인 애플리케이션을 위해 아래의 원칙들을 엄격히 준수하세요.

## 1. Red, Green, Refactor 사이클 준수
기능을 구현할 때 반드시 다음 3단계 사이클을 반복하십시오.
- **Red (실패하는 테스트 작성)**: 구현할 기능에 대한 요구사항 기반의 테스트 코드를 먼저 작성합니다. 실제 코드가 없으므로 테스트가 실패하는 것을 반드시 직접 확인해야 합니다.
- **Green (테스트 통과를 위한 최소한의 코드 작성)**: 테스트를 통과시키기 위한 가장 단순하고 직관적인 코드를 작성하여 테스트를 통과(Green 상태) 시킵니다.
- **Refactor (코드 개선)**: 테스트가 통과하는 상태를 유지하면서 코드를 클린 아키텍처 원칙에 맞게 리팩토링합니다. 중복을 제거하고 가독성과 구조를 개선합니다.

## 2. 테스트 피라미드 기반의 포괄적인 테스트 작성
단순한 단위 테스트에 그치지 않고, 시스템 전반의 안정성을 위해 다음 세 가지 레벨의 테스트를 모두 작성해야 합니다.
- **Unit Test (단위 테스트)**: Entity, UseCase(Application Layer), Mapper 등 비즈니스 로직과 비동기 함수의 동작을 검증합니다. 빠른 실행이 목적입니다.
- **Integration Test (통합 테스트)**: UI 컴포넌트 간 상호작용 및 API(Route Handlers, Server Actions) 연동, DB(Repository) 동작이 올바른지 확인합니다.
- **E2E Test (단대단 테스트)**: 브라우저와 연동하여 사용자 관점에서의 전체 흐름(예: 로그인 -> 상품 목록 조회 -> 장바구니 담기 -> 결제)을 검증합니다.

## 3. 구현 후 전체 테스트 실행 및 검증
- 모든 요구사항 구현이 끝난 후 전체 테스트 스위트를 다시 실행합니다.
- 작성한 테스트뿐만 아니라 기존에 존재하던 모든 테스트도 **100% 통과**하는지 확인해야 합니다. 이를 통해 사이드 이펙트(회귀 버그)를 방지합니다.

## 4. 실패한 테스트의 철저한 분석 및 수정
- 단 하나라도 통과하지 못한 테스트가 있다면, 단순히 테스트를 지우거나 코드를 조작하여 덮어쓰지 마세요.
- **왜 테스트가 통과하지 못했는지 상세한 원인을 파악**(로그, 디버깅)하고, 분석된 원인을 토대로 근본적인 로직을 수정한 뒤 다시 테스트를 실행합니다.

## 5. 80% 이상의 테스트 커버리지 리포트 유지
- 프로젝트 전체의 테스트 코드 커버리지는 항상 **최소 80% 이상**으로 유지해야 합니다.
- 테스트 실행 시 커버리지 지표를 확인하여, 누락된 에지 케이스(Edge Cases), 조건 분기 코드 등이 테스트되도록 보완하세요.
